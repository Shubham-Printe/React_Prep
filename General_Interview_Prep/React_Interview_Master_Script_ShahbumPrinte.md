Our frontend team is small — three developers — and I lead the frontend planning, implementation strategy, and reviews.  

When I joined, the project was in trouble because a previous contractor had failed to deliver. I was asked to rescue it by building a minimum viable product that could be demoed to the client. We managed to deliver that and retain the project, which we’ve now scaled significantly.

## ⚙️ 3️⃣ Technical Foundations (React + JS + TS)
I have a strong command of React fundamentals — especially rendering behavior, hooks, and component optimization.  
I rely heavily on React.memo, useCallback, and useMemo where necessary — but I’m equally cautious about not over-optimizing or breaking referential equality.  

For state management, I use Redux Toolkit for scalable global state and Context for local cross-component communication. I also design stores with TypeScript types upfront to ensure safety and clarity.  

I’m comfortable working with SSR through Next.js — including routing, data fetching, and performance optimization for SEO and initial load.  

On the TypeScript side, I care a lot about maintainable types — especially when working with APIs or reusable hooks. I use utility types, generics, and discriminated unions to make code both safe and self-documenting.

**Hook Lines:**
- “Not over-optimizing re-renders.”
- “Using TypeScript-first design to reduce bugs.”
- “SSR for SEO and performance.”

## 🧠 4️⃣ Problem-Solving & Trade-Offs
One thing I’ve learned as a senior frontend developer is that every technical decision is a trade-off.  
For example, we once had to decide between building a complex dashboard from scratch or using a third-party library. Building it ourselves would mean better flexibility, but slower delivery. Using a library was faster but limited customization. I proposed a hybrid — start with the library for delivery speed, and progressively replace modules with our custom ones once the MVP was live. That approach worked really well.  

Similarly, when performance and readability conflict, I prefer clear code first — optimize only when profiling indicates a real issue. This mindset has saved the team from premature complexity many times.

**Hook Lines:**
- “Hybrid approach — start with library, then customize later.”
- “Performance vs readability trade-off.”
